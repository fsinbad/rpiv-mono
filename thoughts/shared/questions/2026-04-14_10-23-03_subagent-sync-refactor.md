# Research Questions: Session-start subagent copy implementation for upcoming add/remove/edit sync refactor

## Discovery Summary
Discovery showed the current sync logic is centered in `extensions/rpiv-core/agents.ts` and invoked from two paths in `extensions/rpiv-core/index.ts`: automatic `session_start` (`overwrite=false`) and manual `/rpiv-update-agents` (`overwrite=true`). The implementation currently enumerates bundled `agents/*.md`, copies missing or all files depending on mode, and never removes stale destination files in `<cwd>/.pi/agents/`. Key files read for depth were `extensions/rpiv-core/index.ts`, `extensions/rpiv-core/agents.ts`, `extensions/rpiv-core/guidance.ts`, `extensions/rpiv-core/package-checks.ts`, `package.json`, `README.md`, plus historical artifacts in `thoughts/shared/questions/2026-04-13_15-53-01_agent-resolution-in-plugin.md` and `thoughts/shared/designs/2026-04-10_11-18-29_complete-pi-migration.md`. Historical design intent explicitly favored skip-if-exists on startup and explicit overwrite via command, which now must be revisited against the new add/remove/edit detection requirement. The area is small but cross-cutting: utility-level filesystem behavior in `agents.ts` drives user-visible command UX and runtime subagent resolution in `@tintinweb/pi-subagents`.

## Questions

1. Trace the end-to-end session bootstrap path that currently performs non-destructive agent sync: `package.json` (`pi.extensions`) loads `extensions/rpiv-core/index.ts` default export, `pi.on("session_start")` invokes `copyBundledAgents(ctx.cwd, false)`, and `copyBundledAgents` in `extensions/rpiv-core/agents.ts` enumerates `BUNDLED_AGENTS_DIR` with `readdirSync`, guards existing targets via `existsSync(dest)`, then writes only missing files with `copyFileSync`. Include how this hook coexists with adjacent startup concerns in `index.ts` (guidance injection, thoughts scaffolding, git context, missing-package warnings) and why those neighboring side effects constrain where refactor logic can safely throw or block. Name the concrete return contract `{ copied: string[]; skipped: string[] }` and where it is used for UI notifications. Explain why this path is the baseline for “add-only unless missing” behavior and what invariants must be preserved.

2. Trace the manual refresh path registered by `pi.registerCommand("rpiv-update-agents")` in `extensions/rpiv-core/index.ts`, including how the command handler calls `copyBundledAgents(ctx.cwd, true)`, how overwrite mode bypasses the `existsSync(dest)` skip branch in `agents.ts`, and how `ctx.ui.notify` reports refresh results. Contrast this with session-start semantics to show where local user edits are currently preserved vs forcibly replaced. Identify exactly which function boundary should absorb new change-detection metadata (e.g., edited/unchanged/removed) so command UX can evolve without duplicating filesystem logic in `index.ts`. Explain why this distinction matters for the upcoming refactor’s expected behavior (“replace only changed bundled files and remove no-longer-bundled files”).

3. Trace the source-of-truth resolution for bundled subagent files in `extensions/rpiv-core/agents.ts`: `PACKAGE_ROOT` derived from `fileURLToPath(import.meta.url)` and `dirname(dirname(dirname(...)))`, `BUNDLED_AGENTS_DIR = join(PACKAGE_ROOT, "agents")`, and source listing filtered to `*.md`. Connect that path logic to actual bundled profile assets under `agents/*.md` and documented operational expectations in `README.md` (“editable at <cwd>/.pi/agents/; refresh via /rpiv-update-agents”). Explain edge-case behavior when source dir is absent (early empty result), and evaluate whether the upcoming refactor should keep this silent posture or surface warnings. Include the import chain from `index.ts` to `agents.ts` and why isolating path resolution in a pure utility simplifies future diff strategies.

4. Define a precise code-path proposal for add/remove/edit detection inside `copyBundledAgents` without violating current architectural boundaries from `extensions/rpiv-core (architecture.md)` (utility module, no `ExtensionAPI`, safe defaults, no throws). The analysis should map where to add destination enumeration (`readdirSync(targetDir)`), how to compute source/dest sets by filename, and where to introduce content comparison (hash or direct buffer compare) before deciding copy/skip/replace. It should also identify the missing stale-cleanup path (currently no `unlinkSync`/`rmSync`) and propose deterministic removal semantics for files present in `.pi/agents` but absent in bundled `agents/`. Specify the evolved result type (e.g., `added`, `updated`, `unchanged`, `removed`, `skipped`) and how to preserve backward compatibility for existing callers in `index.ts`.

5. Trace downstream runtime consumption of `.pi/agents` by `@tintinweb/pi-subagents` using discovery findings captured in `thoughts/shared/questions/2026-04-13_15-53-01_agent-resolution-in-plugin.md`: `loadCustomAgents(cwd)` reads `<cwd>/.pi/agents`, project agents override global/default maps, and `reloadCustomAgents()` is called on each `Agent` tool execution. Explain why this means filesystem sync correctness directly affects live subagent resolution without requiring session restart, and how filename-stem identity rules in that pipeline raise risk if removal behavior is wrong. Connect these runtime guarantees back to the refactor goal so that add/remove/edit detection is validated against actual resolver behavior rather than only file-copy outcomes. Include implications for preserving user-authored local-only agent files if policy intends to keep custom additions.

6. Reconcile intent vs implementation by tracing historical migration/design context in `thoughts/shared/designs/2026-04-10_11-18-29_complete-pi-migration.md` (Decision 5: session-start skip-if-exists + forced overwrite command) against current `README.md` command docs and the present `index.ts`/`agents.ts` behavior. Identify which assumptions from the migration era are now outdated given the new requirement to detect add/remove/edit automatically, and which contracts must remain stable (session hook timing, command existence, destination path, non-crashing startup). Then define acceptance criteria that can guide implementation and manual verification: what counts as “edited,” what gets removed, how user feedback should be phrased, and how to avoid regressions in first-session bootstrap and existing local customizations.
