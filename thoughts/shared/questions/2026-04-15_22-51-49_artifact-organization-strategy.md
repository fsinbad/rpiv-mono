# Research Questions: Artifact Organization Strategy

## Discovery Summary
Six parallel discovery agents located 40+ relevant files across rpiv-pi and rpiv-skillbased: the full `thoughts/shared/` tree (61 artifacts across 5 subdirectories), the `.rpiv/` shadow tree (guidance + test-cases), all 17 SKILL.md files with their artifact input/output contracts, the `thoughts-locator`/`thoughts-analyzer` agents that traverse artifacts, and the session-hooks scaffolding in `extensions/rpiv-core/session-hooks.ts`. Ten key files were read for depth including the architecture guidance, GUIDE.md pipeline definitions, the rpiv-pi-state-audit research (which found 16 dead artifacts in 5 days), the skill-flow-chaining research (establishing disk artifacts as the inter-skill contract), the pi-claude-md-subfolder-resolution research (establishing the `.rpiv/guidance/` vs CLAUDE.md priority ladder), the naming-conventions-unification research, and the `create-handoff` SKILL.md template. No existing documents address external tool integration (Linear/Jira) — this is entirely unexplored territory.

## Questions

1. Trace the current physical separation between `thoughts/shared/` and `.rpiv/`: `session-hooks.ts:22-28` scaffolds five `thoughts/shared/{type}/` directories at session start via `mkdirSync`; `.rpiv/guidance/` holds architecture docs injected by `guidance.ts:52-97` on `tool_call`; `.rpiv/test-cases/` holds QA artifacts produced by `outline-test-cases` and `write-test-cases`. The `thoughts/` tree is gitignored in rpiv-skillbased but committed in rpiv-pi, and the `.rpiv/` tree is always committed. Analyze the implicit organizing principle — is it "ephemeral research vs permanent project metadata"? If so, the current split is inconsistent: pipeline artifacts in `thoughts/` like plans and designs often outlive their session and get referenced by `resume-handoff` (SKILL.md line 18) across sessions. Evaluate whether moving pipeline artifacts under `.rpiv/artifacts/` or `.rpiv/pipeline/` would better express their lifecycle, what breaks in the 17 SKILL.md files that hard-code `thoughts/shared/` paths, and whether a third location (e.g., project root `artifacts/`) avoids overloading `.rpiv/` which currently serves tool-specific infrastructure.

2. Currently artifacts are grouped by type (`questions/`, `research/`, `designs/`, `plans/`) with topic threads emerging organically from matching kebab-case slugs across directories — e.g., `complete-pi-migration` appears in all four type directories as shown in the discovery summary. The `thoughts-locator` agent at `agents/thoughts-locator.md:118-121` traverses these threads by following `questions_source`, `research_source`, `design_source` frontmatter fields. Compare three grouping strategies: (A) current type-first with implicit topic threads, (B) feature-first grouping like `thoughts/shared/features/complete-pi-migration/{questions,research,design,plan}.md`, (C) hybrid with a feature index that cross-references type-organized files. For each, analyze the impact on: the 7 skill producers that write `thoughts/shared/{type}/YYYY-MM-DD_HH-MM-SS_topic.md` filenames, the `thoughts-locator` and `thoughts-analyzer` agents that grep frontmatter fields, the chaining instructions in each SKILL.md that emit next-skill commands with artifact paths, and the `create-handoff`/`resume-handoff` pair that lists artifacts exhaustively (SKILL.md line 62-67).

3. The current cross-referencing system has three layers: (a) forward chain instructions in SKILL.md prose ("run `/rpiv-next:research thoughts/shared/questions/[filename].md`"), (b) backward `*_source` frontmatter fields (`questions_source`, `research_source`, `design_source`) set by `research` (SKILL.md:190), `design` (SKILL.md:262), and `write-plan` (SKILL.md:118), and (c) inline `## References` sections in the artifact body. There is no index file, no manifest, and no bidirectional graph — finding all downstream artifacts from a questions doc requires grepping all files for its path. The test-case system uses `_meta.md` as an inter-skill contract (`outline-test-cases/SKILL.md:397`) and `_coverage-map.md` as a project-wide index (`write-test-cases/SKILL.md:276`). Evaluate whether a per-topic index file (e.g., `thoughts/shared/index/complete-pi-migration.md` listing all artifacts in the chain), a global manifest (e.g., `thoughts/shared/manifest.json` with artifact metadata), or enriched frontmatter (adding `produces` forward pointers alongside existing `*_source` backward pointers) would enable better navigation — and what the maintenance cost is for each when skills already struggle with stale references (research audit Q2 found ~20 stale guidance references).

4. The rpiv-pi-state-audit research (`2026-04-15_21-11-56`) found 16 dead or rejected artifacts accumulated over just 5 days (61 total .md files): 6 plans for work done in sibling repos, 1 abandoned pipeline feature, 1 rejected subagent discipline plan, 2 misfiled sibling-repo artifacts, and 3 duplicate question files for the same topic (`rework-readme-professional-setup`). The `status` frontmatter field exists on all artifacts but has no lifecycle enforcement — values include `complete`, `ready`, `in-progress`, `draft`, yet nothing transitions status to `archived`, `superseded`, or `dead`. The `create-handoff` skill (SKILL.md:66-67) lists artifacts exhaustively but doesn't mark referenced artifacts as "in-use" vs "historical". Analyze what artifact lifecycle management should look like: automatic archival (e.g., move to `thoughts/shared/archive/` after N days or after plan validation), status transitions triggered by downstream skill completion, or a cleanup skill that audits and proposes deletions — considering that the `thoughts-locator` agent treats all artifacts equally regardless of status.

5. No existing documents, code, or configuration address integration with external project management tools — this is entirely unexplored territory per the thoughts-locator discovery. Current artifact frontmatter has fields that partially map to issue tracker concepts: `topic` → ticket title, `tags` → labels, `status` → workflow state, `date` → created date, `branch` → linked branch. However, the mapping is lossy: there's no `epic`/`project` field for grouping, no `assignee` beyond `researcher`/`designer`/`planner`, no priority, no external ID field for bidirectional sync. Analyze what metadata additions would enable future integration without over-engineering: should artifacts carry an optional `external_id` or `linear_id` field? Should the feature-grouping mechanism (question 2) double as the epic/project mapping? What's the minimal frontmatter schema extension that lets a future integration skill or MCP server sync artifact status bidirectionally with Linear/Jira while preserving the current pipeline's simplicity?

6. Current artifacts are monolithic markdown files — a research document can be 400+ lines (e.g., `rpiv-pi-state-audit` research is 403 lines) covering 10+ questions with distinct sections. Skills consume entire artifacts: `design` reads the full research artifact, `write-plan` reads the full design. But the `design` skill (SKILL.md:3) specifies it works on "complex multi-component features touching 6+ files across multiple layers" using "iterative vertical-slice decomposition" — each slice is conceptually a sub-artifact within the design. The `iterate-plan` skill edits plans in-place (SKILL.md:133) rather than producing new artifacts, creating implicit versioning with no history. Evaluate mechanisms for sub-artifact referencing: markdown heading anchors (`research.md#q3-guidance-resolution`), splitting large artifacts into directories (`designs/pi-migration/{slice-1.md, slice-2.md, index.md}`), or keeping monolithic files but adding a `## Sub-Artifact Index` section with stable IDs — weighing discoverability, skill complexity, and whether the `thoughts-locator`/`thoughts-analyzer` agents can navigate sub-artifacts without major changes.

7. The test-case pipeline uses a distinctly different organization than `thoughts/shared/`: feature-slug directories (`.rpiv/test-cases/{feature-slug}/`), `_meta.md` as inter-skill contract with structured frontmatter (`feature`, `module`, `portal`, `slug`, `status` per `outline-test-cases/templates/feature-meta.md:3-10`), individual test case files with IDs (`TC-{MODULE}-{NNN}`), `_regression-suite.md` per feature, and `_coverage-map.md` as project-wide rollup (`write-test-cases/SKILL.md:276`). This is feature-first grouping with rich metadata — the opposite of `thoughts/shared/`'s type-first flat files. Evaluate whether this pattern could be adapted for pipeline artifacts: `thoughts/shared/{feature-slug}/{questions.md, research.md, design.md, plan.md, _meta.md}` with a `_meta.md` tracking pipeline progress, status, and cross-references. What would break in the timestamp-based filename convention, the `THOUGHTS_DIRS` scaffolding in `session-hooks.ts:22-28`, and the 7 producer skills that currently write to fixed type directories?

8. `session-hooks.ts:22-28` scaffolds `thoughts/shared/{research,questions,designs,plans,handoffs}` at every `session_start`, treating these as session infrastructure. But artifacts accumulate across sessions indefinitely — the skill-flow-chaining research (`2026-04-12`) established that "only files on disk survive a session reset" and disk artifacts are "the inter-skill contract." The `create-handoff`/`resume-handoff` pair bridges sessions by listing artifact paths, but doesn't distinguish current-session artifacts from historical ones. Meanwhile, `GUIDE.md:159-173` documents a `thoughts/me/` personal directory and `thoughts/global/` cross-repo directory that exist in agent definitions but are never scaffolded by `session-hooks.ts` and have no producer skills. Analyze the tension between session-scoped creation (each session generates new timestamped artifacts) and project-scoped consumption (artifacts are referenced across sessions, searched by agents, and cleaned up manually). Should the scaffolding create per-session subdirectories? Should there be an explicit "workspace" or "active feature" concept that scopes which artifacts are visible to skills? How does this interact with the `thoughts/me/` and `thoughts/global/` directories that are defined but apparently unused?
