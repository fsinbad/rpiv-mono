# Research Questions: `/btw` Side-Question Slash Command Extension

## Discovery Summary

The `/btw` command is a sibling Pi extension to `@juicesharp/rpiv-advisor` that exposes a user-invoked slash command for asking the same primary model a one-off side question with the cloned primary conversation as context. Discovery covered the blueprint extension at `/Users/sguslystyi/rpiv-advisor/` (28-line `index.ts`, 488-line `advisor.ts`, 137-line `advisor-ui.ts`, single-file `prompts/advisor-system.txt`, 37-line `package.json`), the host package `rpiv-pi/extensions/rpiv-core/` (orchestrator with `session-hooks.ts`, `setup-command.ts`, `siblings.ts`), and the upstream Pi host packages â€” `@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts` (1074 lines defining `ExtensionAPI`, `ExtensionContext`, `ExtensionCommandContext`, all event types), `@mariozechner/pi-coding-agent/dist/core/messages.d.ts` (`convertToLlm`, `CustomMessage`), `@mariozechner/pi-coding-agent/dist/core/compaction/utils.js` (`serializeConversation` formatter), `@mariozechner/pi-ai/dist/types.d.ts` (`Message`, `Context`, `AssistantMessage`, `StopReason`), `@mariozechner/pi-ai/dist/stream.d.ts` (`completeSimple`, `streamSimple`), and `@mariozechner/pi-coding-agent/docs/prompt-templates.md` (the user-facing prompt template loader). The user-supplied screenshot of the `/btw` system reminder names `SIDE_QUESTION` as the only template variable and asserts "main agent is NOT interrupted â€” it continues working independently in the background", which constrains the concurrency model. Advisor-the-tool is registered via `pi.registerTool` and invoked by the LLM; `/btw` differs in that it is invoked by the *user* via `pi.registerCommand`, takes its question as command args, and never registers a tool â€” so the `before_agent_start` strip-tool hook and the persisted-model state machine in advisor are likely unnecessary and the new extension can be substantially smaller than the 488-line advisor module.

## Questions

1. Trace the full `/btw <question>` request path end-to-end, from command registration through LLM round-trip to user-visible output. Walk from the `pi.registerCommand("btw", { handler })` call (mirroring `registerAdvisorCommand` in `rpiv-advisor/advisor.ts:402`), through the handler signature `(args: string, ctx: ExtensionCommandContext) => Promise<void>` (typed in `pi-coding-agent/dist/core/extensions/types.d.ts:693`), into the construction of a one-shot `Context` (`{systemPrompt, messages}` from `pi-ai/dist/types.d.ts:150`) passed to `completeSimple(model, context, {apiKey, headers, signal, reasoning})` (declared in `pi-ai/dist/stream.d.ts:7`), and back out through a render path. Identify every required intermediate â€” `ctx.modelRegistry.getApiKeyAndHeaders`, `ctx.signal`, `ctx.hasUI` guard, error branches for `stopReason==="aborted" | "error"`. This trace defines the entire critical path the new extension must implement.

2. How does `rpiv-advisor` clone the primary conversation context into a fresh LLM call, and what does `/btw` need to reuse vs. change? Trace from `ctx.sessionManager.getBranch()` in `advisor.ts:249` (returning `SessionEntry[]` per `pi-coding-agent/dist/core/session-manager.d.ts:244`), through the `.filter((e): e is SessionEntry & {type:"message"} => e.type === "message").map(e => e.message)` pattern that extracts `AgentMessage[]`, into `convertToLlm(agentMessages)` (exported from `pi-coding-agent/dist/core/messages.d.ts:76`), into `serializeConversation(messages)` (defined in `pi-coding-agent/dist/core/compaction/utils.js:93` â€” emits `[User]:`, `[Assistant]:`, `[Assistant tool calls]:` lines with 2000-char tool-result truncation), and finally into the wrapped `userMessage` containing `## Conversation So Far\n\n${conversationText}` at `advisor.ts:255-264`. Decide whether `/btw` sends the *serialized* string (advisor pattern, history is data not conversation) or the *raw `Message[]`* directly (so the side LLM perceives a real conversation continuation), since the system-prompt screenshot says the side agent "shares the conversation context" â€” the choice changes prompt engineering and token cost.

3. How is the `/btw` system prompt loaded and how does the `$SIDE_QUESTION` template variable get substituted? Compare the simple file-load pattern in `advisor.ts:137-140` (`readFileSync(fileURLToPath(new URL("./prompts/advisor-system.txt", import.meta.url)), "utf-8").trimEnd()` â€” eager module-init load, no variables) against the prompt-templates system documented in `pi-coding-agent/docs/prompt-templates.md` (filename-derived `/name`, frontmatter `description`, `$1`/`$@`/`${@:N:L}` positional args, autoloaded from `prompts/` dirs). The screenshot the user provided is a Markdown frontmatter file (`name`, `description`, `ccVersion`, `variables: [SIDE_QUESTION]`) wrapping a `<system-reminder>` block â€” but pi-coding-agent's prompt-templates layer expands user-typed slash invocations into editor text, *not* the system prompt of an internal LLM call. Determine whether `/btw` should: (a) load `prompts/btw-system.txt` plus do its own `${SIDE_QUESTION}` regex replace before `completeSimple`, (b) reuse pi's prompt-template loader inside the extension to honor the frontmatter, or (c) treat the screenshot's frontmatter as documentation only and inline the `<system-reminder>` constant â€” whichever choice dictates the `prompts/` file format and registration code.

4. Which model should `/btw` invoke and how is it resolved at command time â€” same as primary, configurable override, or both? Contrast advisor's persistent selection (`selectedAdvisor: Model<Api> | undefined` in `advisor.ts:146`, `~/.config/rpiv-advisor/advisor.json` chmod 0600 persistence at `advisor.ts:101-125`, restored on `session_start` via `restoreAdvisorState`, picked via `ctx.modelRegistry.getAvailable()` and `showAdvisorPicker` in `advisor-ui.ts:67`) against the user's stated requirement that `/btw` uses *the same model as primary*. The primary model is exposed at `ctx.model: Model<any> | undefined` (`pi-coding-agent/dist/core/extensions/types.d.ts:192`). Decide whether `/btw` should (a) hard-code `ctx.model` and skip persistence/UI entirely (no `/btw` selector command needed), (b) default to `ctx.model` but allow an optional `/btw-config` to pin a different model (advisor-style picker reuse), or (c) support inline override syntax (`/btw --model=... <question>`). The choice determines whether we need `advisor-ui.ts`-style pickers, `~/.config/rpiv-btw/`, and `restoreXState` at all â€” the simplest path drops 200+ lines vs. advisor.

5. How should the `/btw` LLM response be displayed and persisted â€” ephemeral notification, custom message in-session, or rich UI overlay? Inventory the available output sinks: `ctx.ui.notify(message, "info"|"warning"|"error")` (`extensions/types.d.ts:63`) shows a transient toast; `pi.sendMessage({customType, content, display, details}, {triggerTurn?, deliverAs?})` (`extensions/types.d.ts:751`) appends a `CustomMessage` entry to the session (`messages.d.ts:32`) that round-trips through `convertToLlm` if `display:true` and persists in the session file; `pi.registerMessageRenderer<T>(customType, renderer)` (`extensions/types.d.ts:749`, used nowhere in advisor) lets the extension control TUI rendering; `ctx.ui.custom<T>(factory, {overlay, overlayOptions})` (`extensions/types.d.ts:93-103`) shows a focused component (used by advisor's `showAdvisorPicker` in `advisor-ui.ts:71`). Decide: should the `/btw` answer (a) be a one-shot `notify` (lossy, can't scroll back), (b) be appended via `sendMessage` with `display:true` so it scrolls in the transcript and persists across restarts (and if so, with `customType:"btw_answer"` and a registered renderer for nice formatting), (c) be appended with `display:false` so it never enters the LLM context (preserves the "side conversation, main agent not interrupted" semantics in the system-reminder screenshot), or (d) open a bordered `ui.custom` overlay like advisor's picker until the user dismisses it. Each choice has implications for transcript noise, replay, and whether the main agent ever "sees" the answer on next compaction.

6. Should `/btw` use `completeSimple` (single-shot) or `streamSimple` (token streaming), and how does cancellation propagate from the user pressing Esc? Both are exported from `pi-ai/dist/stream.d.ts:6-7` with the same `(model, context, options?)` signature; advisor uses `completeSimple` at `advisor.ts:272-276` and forwards `signal: ctx.signal` (the active `AbortSignal | undefined` from `extensions/types.d.ts:196`), then handles `response.stopReason === "aborted"` at `advisor.ts:278-289` returning `ERR_CALL_ABORTED`. For `/btw`, decide whether the user-perceived latency of waiting for a full single-shot answer is acceptable (advisor's call is a *tool*, hidden from the user behind a "Consultingâ€¦" status) or whether streaming token-by-token via `streamSimple` plus per-event UI updates (advisor's `onUpdate` callback pattern at `advisor.ts:266-269` is for tool-call partial results â€” different mechanism) is required for UX. Also decide where the abort signal comes from: `ctx.signal` is undefined when the agent is idle, so `/btw` likely needs its own `AbortController` wired to `ctx.ui.onTerminalInput` for Esc, or to `ctx.abort()` (`extensions/types.d.ts:198`).

7. What concurrency rules apply when the user types `/btw` while the primary agent is mid-stream â€” does the command run immediately, queue, or block, and how does that match the system-reminder's "main agent is NOT interrupted" claim? The `ExtensionCommandContext` (`extensions/types.d.ts:214`) extends `ExtensionContext` with `waitForIdle()`, `newSession()`, `fork()`, etc., implying commands can be invoked while streaming. `ctx.isIdle()` (`types.d.ts:194`) reports whether the agent is streaming and `ctx.hasPendingMessages()` (`types.d.ts:200`) reports queued messages. Determine: (a) does pi's interactive mode dispatch slash commands synchronously while a turn is in flight, or does it queue them for after `waitForIdle`, (b) if `/btw` runs concurrently it must spawn its own LLM call without disturbing the main agent's request â€” verify that `completeSimple` against `ctx.model` while the same model is mid-stream is safe (separate HTTP request, separate provider auth header), (c) if it can't run concurrently, the system-reminder's "main agent continues working independently in the background" framing is misleading and must be reworded, and (d) confirm that `/btw`'s own work must NOT call `pi.sendMessage` with `triggerTurn:true` or use any side-effects that would re-enter the main agent loop.

8. What is the exact package layout, manifest, and lifecycle wiring `/btw` must mirror from `rpiv-advisor` to be installable as a sibling extension via `pi install npm:@juicesharp/rpiv-btw`? Reproduce the structural skeleton: `package.json` with `"type":"module"`, `"pi": {"extensions": ["./index.ts"]}`, `peerDependencies` on `@mariozechner/pi-ai`, `@mariozechner/pi-coding-agent`, `@mariozechner/pi-tui` (only if `ui.custom` panels are used), `@sinclair/typebox` (only if registering tools â€” likely *not* needed for `/btw` since no `registerTool` call), `"files": ["index.ts","btw.ts","prompts/","README.md"]` (mirroring `rpiv-advisor/package.json:20-26`); a thin `index.ts` that imports registrar functions and calls them inside `export default function(pi: ExtensionAPI)` (mirroring `rpiv-advisor/index.ts:20-28`); a single `btw.ts` housing constants, system-prompt module-init load, the `registerBtwCommand(pi)` function, and the `executeBtw(ctx, args, signal)` core. Also decide which lifecycle hooks to register: advisor uses `session_start` (restore persisted model) and `before_agent_start` (strip tool when no model selected) â€” `/btw` likely needs *neither* if it's a pure command with no tool, no persisted state, and uses `ctx.model` directly. Finally, confirm the sibling-registration handshake: `rpiv-pi/extensions/rpiv-core/siblings.ts` would need to list `@juicesharp/rpiv-btw` so `/rpiv-setup` installs it (verify in `rpiv-pi/extensions/rpiv-core/package-checks.ts:findMissingSiblings`).
