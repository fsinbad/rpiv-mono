# Research Questions: Extract QA Test-Case Skills into Optional Sibling Plugin

## Discovery Summary

Six discovery agents scanned 18+ files across skills, agents, extensions, package manifests, and prior design artifacts. The test-case subsystem is a cleanly bounded, self-contained pair of skills (`outline-test-cases` with 3 files, `write-test-cases` with 9 files) plus one agent (`test-case-locator.md`) that form a closed loop — zero other skills, extensions, or session hooks reference them. Prior extraction work (2026-04-13 design for 4 tool plugins) established the sibling plugin pattern, and the artifact organization research (2026-04-15) explicitly noted test-case skills "will be extracted to a separate sibling plugin." Eight key files were read for depth: `siblings.ts`, `agents.ts`, `session-hooks.ts`, `package-checks.ts`, `index.ts`, `test-case-locator.md`, the prior extraction design, and the artifact organization research.

## Questions

1. Trace how a new sibling plugin package would be structured — from the `SiblingPlugin` interface in `extensions/rpiv-core/siblings.ts` (lines 13-19: `pkg`, `matches`, `provides` fields), through the `SIBLINGS` array (line 22-48) that feeds presence detection in `package-checks.ts` (`findMissingSiblings()` reads `~/.pi/agent/settings.json`), session-start warning in `session-hooks.ts` (`warnMissingSiblings()` at line 113), and `/rpiv-setup` installer in `setup-command.ts` (`installMissing()` at line 70). The prior extraction design at `thoughts/shared/designs/2026-04-13_17-00-00_extract-rpiv-plugins.md` established the pattern of each sibling owning its own `default export(pi: ExtensionAPI)`, `peerDependencies` on Pi core, and zero cross-plugin runtime imports. This matters because a new `@juicesharp/rpiv-test-cases` sibling would follow this exact registration path: one `SIBLINGS` entry + one `peerDependencies` entry + its own `index.ts` extension — OR it could be made truly optional by omitting it from `SIBLINGS` entirely, requiring investigation into the implications of each approach.

2. Trace how `test-case-locator.md` agent is synced and whether extraction requires a second agent-syncing extension — from `extensions/rpiv-core/agents.ts` where `BUNDLED_AGENTS_DIR` (line 33) resolves to `<PACKAGE_ROOT>/agents/`, through `syncBundledAgents()` (line 156) which enumerates all `.md` files in that directory, copies new ones to `<cwd>/.pi/agents/`, and tracks managed files via `.rpiv-managed.json` manifest. Currently `test-case-locator.md` is one of 9 agents that sync uniformly with no special handling. If extracted to a sibling plugin, the new package would need its own `agents/` directory and its own agent-sync extension — or a mechanism to share the sync infrastructure. This matters because the current sync is `PACKAGE_ROOT`-bound and cannot be driven from two packages simultaneously without coordination.

3. Trace how `outline-test-cases` and `write-test-cases` skills reference the `test-case-locator` agent and whether skills can be split across packages while maintaining agent discoverability — from `skills/outline-test-cases/SKILL.md:58` which spawns `subagent_type: "test-case-locator"` and `skills/write-test-cases/SKILL.md:83` which does the same, through Pi's agent resolution which reads `.pi/agents/` in the current working directory (populated by `syncBundledAgents()`). The `test-case-locator` agent at `agents/test-case-locator.md` uses only `tools: grep, find, ls` and is isolated — it has no dependency on rpiv-pi runtime code. This matters because if the skills and agent move to a sibling, Pi's `"skills": ["./skills"]` discovery in `package.json` would need the sibling to declare its own skills directory, and the sibling would need to sync its own agent to `.pi/agents/`.

4. Trace the inter-skill contract between `outline-test-cases` and `write-test-cases` — from `skills/outline-test-cases/templates/feature-meta.md` which defines the `_meta.md` schema (frontmatter: `feature`, `module`, `portal`, `slug` + sections: Routes, Endpoints, Scope Decisions, Domain Context, Test Data Requirements, Checkpoint History), through `skills/write-test-cases/SKILL.md:26-39` which reads `_meta.md` as warm-start input, to `skills/write-test-cases/SKILL.md:252` which updates `_meta.md` frontmatter (`tc_count`, `status`, `generated`). The templates at `skills/outline-test-cases/templates/outline-readme.md` and `skills/write-test-cases/templates/{test-case.md, regression-suite.md, coverage-map.md}` are runtime-read by skills via the `Read` tool. This matters because extraction must keep these template files co-located with their skills — the inter-skill contract via `_meta.md` is file-format based and works across package boundaries, but template paths are resolved relative to the skill's directory.

5. Trace how Pi discovers skills from multiple packages simultaneously — from `package.json:21-24` where `"pi": { "skills": ["./skills"] }` declares a single local skills directory, through Pi's extension host which loads each installed package independently (each package declares its own `"pi"` field with its own `"skills"` path). The prior extraction design established that each sibling plugin has its own `package.json` with `"pi": { "skills": ["./skills"] }` and Pi merges all installed packages' skill directories into the available skill set. This matters because a `@juicesharp/rpiv-test-cases` sibling would declare its own skills — `outline-test-cases` and `write-test-cases` — and they would appear alongside rpiv-pi's pipeline skills (discover, research, design, etc.) without conflict.

6. Trace how making test-cases optional would affect the session-start experience and other skills — from `session-hooks.ts:39-51` where `session_start` runs `warnMissingSiblings()` (which reads `SIBLINGS` from `siblings.ts`), through the integration-scanner findings that confirmed zero other skills reference `outline-test-cases` or `write-test-cases` (they are a closed-loop pair). The research artifact at `thoughts/shared/research/2026-04-15_23-28-56_artifact-organization-strategy.md` explicitly states test-case skills "will be extracted to a separate sibling plugin." This matters because making the plugin optional means removing it from `SIBLINGS` (so no missing-plugin warning) and using `peerDependencies` with `"optional": true` or omitting it entirely — but then `/skill:outline-test-cases` would simply be unavailable without a user-facing explanation unless a graceful fallback exists.

7. Trace what files would be removed from rpiv-pi and what would remain — the extraction inventory is: `skills/outline-test-cases/` (3 files: SKILL.md + 2 templates), `skills/write-test-cases/` (9 files: SKILL.md + 3 templates + 5 examples), `agents/test-case-locator.md` (1 file). References to update: `README.md` (3 lines: skill table + agent table), `.rpiv/guidance/skills/architecture.md` (1 line listing test-case skill dirs), `.rpiv/guidance/agents/architecture.md` (1 line listing test-case-locator), `.pi/agents/.rpiv-managed.json` (1 entry), `.pi/agents/CLAUDE.md` (1 line grouping locators). The `extensions/rpiv-core/` code has zero test-case-specific logic — no guidance injection, no session hooks, no command references. This matters because the extraction is cleanly bounded: 13 files move, 5 documentation/manifest files get minor edits, zero TypeScript code changes in rpiv-core.

8. Trace the full package.json shape for the new sibling plugin — using the prior extraction's pattern from `thoughts/shared/designs/2026-04-13_17-00-00_extract-rpiv-plugins.md` where each sibling has `name: "@juicesharp/rpiv-*"`, `keywords: ["pi-package", "pi-extension"]`, `type: "module"`, `files` including `skills/` and `agents/`, `pi: { skills: ["./skills"] }` (no `extensions` needed if no session hooks/commands), and `peerDependencies` on `@mariozechner/pi-coding-agent` and `@tintinweb/pi-subagents`. The sibling would also need `peerDependencies` on `@juicesharp/rpiv-web-tools` if test-case-locator ever uses web tools (it doesn't — only `grep, find, ls`), and potentially on `@juicesharp/rpiv-ask-user-question` since the skills use `ask_user_question` for developer checkpoints. This matters because the peer dependency set determines what the optional plugin transitively requires.

9. Trace the optional-install user experience end-to-end — from a user who runs `pi install npm:@juicesharp/rpiv-pi` and then `/rpiv-setup` (which installs hard-required siblings), through the scenario where test-cases is NOT installed: `/skill:outline-test-cases` is simply absent from the skill menu (Pi doesn't list skills from uninstalled packages), no session-start warning fires (because test-cases is optional, not in `SIBLINGS`), and no guidance or agent sync is affected. A user who wants test-cases would run `pi install npm:@juicesharp/rpiv-test-cases` manually. This matters because the "optional" semantics mean rpiv-pi should NOT list test-cases as a peerDependency (which would trigger npm warnings) and should NOT warn about it being missing — it's a separate, independently installed capability.
