# Research Questions: Study naming conventions for skills, agents, and commands to unify and rename flow semantics

## Discovery Summary
Discovery located naming contracts across runtime TypeScript entrypoints (`extensions/rpiv-core/index.ts`, `extensions/rpiv-core/agents.ts`, `extensions/rpiv-core/package-checks.ts`), skill/agent frontmatter conventions, and user-facing command inventories in `README.md` and guidance docs. The workflow seam is currently represented in three coupled layers: command labels (`/skill:*`, `/rpiv-*`), skill and agent identifiers (`name`, `subagent_type`), and artifact-path semantics (`thoughts/shared/{questions,research,designs,plans}`). Key files were read for depth to trace how names are materialized by runtime hooks (`pi.on`, `pi.registerCommand`), copied into workspace state (`copyBundledAgents`), and referenced in chained skill outputs. Historical thoughts artifacts indicate prior naming migrations and suggest compatibility risks when renaming without dual-name periods. Overall, the codebase exposes multiple partially-overlapping “source of truth” documents, creating a strong need for explicit naming authority and migration choreography.

## Questions

1. Trace the canonical `/skill:<name>` naming contract end-to-end from `package.json` (`pi.skills`) through skill documents and chaining outputs, and identify where a rename must be applied to stay coherent. Start at `package.json` where `"skills": ["./skills"]` exposes the directory to Pi, then follow `.rpiv/guidance/skills/architecture.md` (`name` frontmatter rule) and each `skills/*/SKILL.md` frontmatter `name:` field as the local source-of-truth for invocation identifiers. Continue through user-facing and machine-handoff surfaces in `README.md` workflow examples and the explicit chaining emitters in `skills/research-questions/SKILL.md` and `skills/research/SKILL.md` that print next-step `/skill:...` commands with artifact paths. Include how `description` and `argument-hint` semantics in frontmatter shape user understanding of stage names even when command identifiers are unchanged. This matters because any unification proposal that renames phases (e.g., `research-questions` to `discover`) fails unless manifest-discovery, frontmatter identity, chain emitters, and docs are all updated atomically.

2. Map slash-command naming ownership and prefix strategy from runtime registration in `extensions/rpiv-core/index.ts` to dependency checks in `extensions/rpiv-core/package-checks.ts`, then determine a coherent naming policy for first-party vs sibling-provided commands. Trace concrete registrations via `pi.registerCommand("rpiv-update-agents", ...)` and `pi.registerCommand("rpiv-setup", ...)` in the `export default function (pi: ExtensionAPI)` entrypoint, then follow setup-time messaging that references `/todos`, `/advisor`, and `/web-search-config` as externally provided capabilities. Connect these references to checker functions (`hasRpivTodoInstalled()`, `hasRpivAdvisorInstalled()`, `hasRpivWebToolsInstalled()`, `hasPiSubagentsInstalled()`) and their regex contracts in `package-checks.ts`, plus package peer names in `package.json`. Compare the resulting command namespace shape with the command inventories in `README.md` and `.rpiv/guidance/architecture.md`, highlighting where prefixes (`rpiv-` vs bare names) encode ownership inconsistently. This matters because rename/unification decisions should preserve discoverability and avoid implying wrong ownership boundaries between rpiv-pi core and sibling plugins.

3. Trace named-subagent identity invariants from agent definition files through workspace materialization and skill dispatch, then specify which renames are structurally safe versus breaking. Begin at `agents/*.md` frontmatter `name` values (e.g., `codebase-locator`, `integration-scanner`), then follow `extensions/rpiv-core/agents.ts` (`PACKAGE_ROOT`, `BUNDLED_AGENTS_DIR`, `copyBundledAgents(cwd, overwrite)`) into `extensions/rpiv-core/index.ts` session-start auto-copy and `/rpiv-update-agents` refresh command. Continue into caller surfaces where skills reference `subagent_type` (notably `skills/research-questions/SKILL.md`) and align with constraints in `.rpiv/guidance/agents/architecture.md` requiring filename stem = frontmatter `name` = `subagent_type` literal. Include how `.pi/agents/` copies can drift locally and what that implies for staged rename rollouts. This matters because agent renaming is a three-part identity migration (file, metadata, call sites) and partial changes silently degrade skill behavior by falling back to generic agents.

4. Analyze phase naming semantics in the research→design→plan pipeline by tracing artifact directories created at runtime to the skill contracts that consume and emit those artifacts, and evaluate whether names should encode action, artifact, or sequence. Start in `extensions/rpiv-core/index.ts` where session bootstrap `mkdirSync(join(ctx.cwd, dir), { recursive: true })` materializes `thoughts/shared/questions|research|designs|plans|handoffs`. Then trace how `skills/research-questions/SKILL.md` and `skills/research/SKILL.md` encode “Phase 1/Phase 2” semantics and produce chained commands that pass explicit artifact paths, and compare with the pipeline labels documented in `.rpiv/guidance/architecture.md`, `.rpiv/guidance/skills/architecture.md`, and `README.md`. Identify whether current names reflect verb-first workflow stages, noun-first artifact stores, or mixed conventions, and where this causes cognitive mismatch (for example, skill name vs output-folder pluralization). This matters because a clearer flow language should reduce operator error when manually chaining commands and selecting entry points mid-pipeline.

5. Establish a single-source naming contract by tracing how naming rules are duplicated across guidance, README, and skill docs, then determine which location should be authoritative and how others should be derived or validated. Follow rule definitions in `.rpiv/guidance/skills/architecture.md` (frontmatter schema), `.rpiv/guidance/agents/architecture.md` (agent identity contract), and `.rpiv/guidance/architecture.md` (top-level command/pipeline map), then compare with parallel narrative tables in `README.md` and inline guidance inside `skills/research-questions/SKILL.md` and `skills/research/SKILL.md`. Capture concrete divergence risks: stale chain examples, renamed commands not reflected in setup docs, or stage terms that differ between guidance and user-facing onboarding. Include where runtime code in `extensions/rpiv-core/index.ts` is the executable truth for command names while Markdown is declarative truth for skills and agents. This matters because unification should reduce maintenance overhead by deciding which artifacts are normative and which are generated or consistency-checked.

6. Design a rename migration strategy that preserves operational compatibility by tracing install/bootstrap and compatibility touchpoints from package naming through setup UX and historical artifacts. Start with `package.json` peer dependency names (`@juicesharp/rpiv-*`, `@tintinweb/pi-subagents`), then follow `extensions/rpiv-core/index.ts` `/rpiv-setup` install list and user warnings, and `extensions/rpiv-core/package-checks.ts` detection regexes that gate capability messaging. Add historical precedent context from `thoughts/shared/*` naming-related migration artifacts discovered during thoughts-locator scanning to identify prior break patterns and safe rollout patterns (alias periods, dual-name docs, staged deprecation). Include the interaction with `extensions/rpiv-core/agents.ts` copy semantics and `.pi/agents` local overrides when agent names change. This matters because naming unification is effectively an ecosystem migration; without a compatibility plan, command, skill, and agent renames can strand existing user setups and automation scripts.
