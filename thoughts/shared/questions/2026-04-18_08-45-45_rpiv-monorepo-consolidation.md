# Research Questions: Organize rpiv-pi and Siblings into a Monorepo

## Discovery Summary

Seven `@juicesharp/rpiv-*` sibling repos live side-by-side on disk (`rpiv-pi`, `rpiv-advisor`, `rpiv-ask-user-question`, `rpiv-btw`, `rpiv-todo`, `rpiv-web-tools`, plus the outlier `rpiv-skillbased`) and on GitHub under `github.com/juicesharp/*` — all independent repositories, all shipping raw TypeScript source (loaded at runtime by `@mariozechner/jiti`), all with zero `scripts`, zero `devDependencies`, and no `tsconfig.json`. Cross-sibling runtime coupling is **zero** — not a single `import` statement crosses sibling boundaries. The only binding surfaces are (a) the declarative `SIBLINGS` registry in `extensions/rpiv-core/siblings.ts:22-53` plus its three consumers (`package-checks.ts:30-33`, `setup-command.ts:32-95`, `session-hooks.ts` `warnMissingSiblings()`), and (b) the `peerDependencies: "*"` block in `rpiv-pi/package.json:25-33`. The external reference — `badlogic/pi-mono` (publishes `@mariozechner/*`) — uses plain **npm workspaces** (no Turbo/Nx/Lerna), lockstep versioning via `scripts/release.mjs` + `scripts/sync-versions.js`, a root `tsconfig.base.json` with per-package `tsconfig.build.json`, Biome for lint+format, and Husky hooks; publishes run locally, not in CI. Key files read for depth: `siblings.ts`, `package-checks.ts`, `setup-command.ts`, `CHANGELOG.md`, and the test-cases research doc (the only prior thought that surfaces the monorepo-vs-separate-repo question, left as an open question).

## Questions

1. Trace the current cross-package binding surface — the `SiblingPlugin` interface and `SIBLINGS` array at `extensions/rpiv-core/siblings.ts:13-53`, its three consumers (`findMissingSiblings()` in `package-checks.ts:30-33`, `buildConfirmBody()` + `installMissing()` in `setup-command.ts:32-95`, `warnMissingSiblings()` in `session-hooks.ts`), the 7-entry `peerDependencies: "*"` block in `rpiv-pi/package.json:25-33`, and the fact that **zero** `import` statements cross sibling boundaries (only `@mariozechner/pi-coding-agent`, `@mariozechner/pi-tui`, `@mariozechner/pi-ai`, `@sinclair/typebox` are imported by siblings). This baseline determines what actually needs rewiring vs what is already loose coupling that survives an unmodified move into `packages/*`.

2. Compare pi-mono's structural template (npm workspaces at root, `packages/*` plus nested example workspaces, `scripts/release.mjs` + `scripts/sync-versions.js` for lockstep release, root `tsconfig.base.json` with per-package `tsconfig.build.json`, root `biome.json`, Husky hooks, `tsgo` builds, Vitest per-package) to the current rpiv-* shape (zero `scripts`, zero `devDependencies`, no `tsconfig.json` anywhere, `LICENSE` only in `rpiv-btw`, ships raw `.ts` via `@mariozechner/jiti`). Name which pi-mono pieces adopt verbatim (npm workspaces, release script skeleton, root Biome, root LICENSE, root `.gitignore`), which adapt (`tsconfig.base.json` for IDE typechecking only, no `tsconfig.build.json` needed), and which are irrelevant because rpiv packages have no build step, no `src/`+`test/` split, no Vitest suites.

3. Evaluate the versioning model — lockstep (pi-mono's `scripts/release.mjs <patch|minor|major>` calls `npm version -ws --no-git-tag-version` to bump every workspace package to a shared version, `sync-versions.js` rewrites inter-package `dependencies`, then a single `git tag v<version>` and `npm run publish` covers the fleet) vs the current independent state (`rpiv-pi@0.6.0`, `rpiv-advisor@0.1.3`, `rpiv-ask-user-question@0.1.4`, `rpiv-btw@0.1.1`, `rpiv-todo@0.1.2`, `rpiv-web-tools@0.1.2` — six diverged lines, only rpiv-pi maintains a CHANGELOG). Identify which SIBLINGS consumers care about versions (the `matches` regex at `siblings.ts:25-51` is version-agnostic and unaffected; `peerDependencies: "*"` is too) and what a lockstep bump means when a bugfix in `rpiv-web-tools` forces an `@juicesharp/rpiv-pi` version bump visible to every downstream user.

4. Decide the inter-package reference protocol once all packages live in one repo — `workspace:*` (npm 7+, cleanest, but risks leaking the `workspace:` specifier into published tarballs unless stripped on publish), pinned semver rewritten on release by a `sync-versions.js` clone (pi-mono's approach, e.g. `"@mariozechner/pi-ai": "0.67.68"` resolved locally via workspace symlinks), or keep `"*"` peers (minimum change, accepts resolution drift). Trace how `/rpiv-setup` flows (`setup-command.ts:43-67` builds the missing list, calls `spawnPiInstall()` in `pi-installer.ts`, which writes to `~/.pi/agent/settings.json` — read by `package-checks.ts:11,14-22`, NOT `node_modules/`), so the monorepo developer's workspace resolution can diverge from the installed user's `~/.pi/agent/settings.json` path without either side breaking.

5. Plan the GitHub consolidation — merge the seven `github.com/juicesharp/rpiv-{pi,advisor,ask-user-question,btw,todo,web-tools,skillbased}` repos into one (`github.com/juicesharp/rpiv-monorepo` or similar, choose name). Compare `git subtree add --prefix=packages/<name>` per source repo (preserves full history under a prefix, simple, produces merge commits) vs `git filter-repo --path-rename` + `git merge --allow-unrelated-histories` (cleaner linear history, more work). Decide tag fate (rpiv-pi carries `v0.6.0`, `v0.5.1`, `v0.5.0`, `v0.4.x`, `v0.3.0`, `v0.2.0`; siblings carry `v0.1.x`), source-repo fate (archive/redirect/delete), and CHANGELOG link rewriting — `rpiv-pi/CHANGELOG.md:62-65` hardcodes `github.com/juicesharp/rpiv-pi/compare/v0.6.0...HEAD` which 404s unless the source repo redirects persist; the npm package name `@juicesharp/rpiv-pi` stays stable, but `package.json:repository.url` in every package flips to the monorepo URL.

6. Decide root-vs-package config ownership. Move to monorepo root: shared `LICENSE` (currently only in `rpiv-btw/LICENSE`), unified `.gitignore` (today diverged across `rpiv-btw`, `rpiv-todo`, `rpiv-skillbased` — the rest have none), `tsconfig.base.json` for IDE typechecking only (no build — we ship raw source), Biome config, Husky, a single root `package.json` with `workspaces` + `scripts` + `devDependencies`. Stay per-package: `README.md`, `CHANGELOG.md`, `package.json` (with `name`, `version`, `pi`, `peerDependencies`, `files`), `prompts/*.txt` (used by `rpiv-advisor`, `rpiv-btw`), `docs/<name>.jpg` (README banner). Skip entirely because we have no build step: `tsconfig.build.json`, `vitest.config.ts`, `dist/`. Call out the `files` array inconsistency (advisor/btw/todo declare `files`; ask-user-question/web-tools omit it and rely on npm defaults) that a shared template must normalize, and decide whether to add missing `files` arrays or rely on a root `.npmignore`.

7. Verify that the `PACKAGE_ROOT`-bound agent-sync logic in `extensions/rpiv-core/agents.ts:27-33` survives a monorepo layout. `PACKAGE_ROOT` is derived from `import.meta.url` + three `dirname()` calls; `BUNDLED_AGENTS_DIR = join(PACKAGE_ROOT, "agents")`; `syncBundledAgents()` at lines 156-267 copies those into `<cwd>/.pi/agents/` using `.rpiv-managed.json` (single flat array, replaced on write — see the test-cases research at `thoughts/shared/research/2026-04-16_11-39-33_extract-test-cases-sibling-plugin.md` for the single-writer constraint). Confirm resolution works under (a) the published tarball at `node_modules/@juicesharp/rpiv-pi/` (unchanged — `files: ["extensions/", "skills/", "agents/", "scripts/", "README.md", "LICENSE"]` at `rpiv-pi/package.json:17` still ships the right subset, **provided** `LICENSE` is either retained per-package or the monorepo root `LICENSE` is copied in during pack), and (b) workspace symlinks from `node_modules/@juicesharp/rpiv-pi/` → `packages/rpiv-pi/` during local dev. If the three-dirname heuristic is fragile, consider `fileURLToPath(import.meta.url)` + explicit walk to a marker file (`package.json`) as the robust alternative.

8. Decide whether `/Users/sguslystyi/rpiv-skillbased` enters the monorepo at all. It's an outlier: different git remote (`repos.truvisibility.com/truvis/rpiv-next.git`, not `github.com/juicesharp`), stub `package.json` (`{ "type": "module" }` only — no `name`, no `version`, no `pi` config, no peers), README titled `# rpiv-next` (branding drift from the folder name), no TypeScript source, and content overlap — the **same 9 agent filenames** as `rpiv-pi/agents/` plus ~13 overlapping skill folders (`annotate-guidance`, `annotate-inline`, `code-review`, `commit`, `design`, `implement-plan`, `iterate-plan`, `migrate-to-guidance`, `outline-test-cases`, `research`, `research-questions`, `research-solutions`, `resume-handoff`, `validate-plan`, `write-plan`, `write-test-cases`). Evaluate three paths: (a) exclude — it's a Claude Code skill bundle with a different audience, not a Pi plugin; (b) include and deduplicate into a shared `@juicesharp/rpiv-skills` package consumed by both Pi and CC hosts; (c) include as `packages/rpiv-skillbased` retaining its separate truvis remote via git submodule or separate branch. Note that including it as-is would fail `npm publish` (no package name), so option (c) implies leaving it unpublished.

9. Choose the publish pipeline. Compare pi-mono's local-run model (maintainer runs `npm run release:patch` on a laptop with clean worktree + npm credentials; `.github/workflows/ci.yml` only builds+tests+checks on push; `build-binaries.yml`, `pr-gate.yml`, `issue-gate.yml` handle bots and binaries but never publish), Changesets (`.changeset/*.md` entries accumulated per PR, `changesets/action` in CI opens a release PR, merging publishes via `NPM_TOKEN` secret, supports independent versioning cleanly), and the status quo (manual per-repo `npm publish` after hand-editing CHANGELOG). Name the trade-offs for a two-person org — pi-mono requires a trusted laptop but no CI secrets; Changesets requires `NPM_TOKEN` + `GITHUB_TOKEN` in the monorepo's GitHub Actions and shifts the "what version?" decision from release-time into PR-time. Trace which release path leaves `rpiv-pi/CHANGELOG.md` in its current hand-edited Keep-a-Changelog format (lines 10-24 are human-authored prose) unchanged — pi-mono's `release.mjs` auto-promotes `## [Unreleased]` to a versioned header, Changesets auto-generates entries from `.changeset/*.md`, status quo leaves it fully manual.
