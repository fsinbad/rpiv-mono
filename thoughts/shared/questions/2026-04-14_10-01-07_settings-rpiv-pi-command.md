# Research Questions: `/settings-rpiv-pi` Command

## Discovery Summary

`rpiv-pi` currently ships two slash commands (`/rpiv-update-agents`, `/rpiv-setup`), both registered inline in `extensions/rpiv-core/index.ts`; there is no settings command and no persistence infrastructure in the repo (zero `writeFileSync` / `saveConfig` / `~/.config/` write sites — only the read-only `readInstalledPackages()` at `package-checks.ts:21-30` against `~/.pi/agent/settings.json`). The user's explicit example — flipping `display: false | true` — lands at five call sites: three `pi.sendMessage({ customType: "rpiv-git-context", display: false })` in `index.ts` (lines 57, 94, 118) and two `pi.sendMessage({ customType: "rpiv-guidance", display: false })` in `guidance.ts` (lines 147-151, 187-191). Locator analysis surfaced ten realistic v1 candidates (the `display` toggles plus `thoughts/shared/*` scaffold paths, `.rpiv/guidance/architecture.md` root, `copyBundledAgents` overwrite flag, notification verbosity, guidance tool-name filter `["read","edit","write"]`, install timeout `120_000`, git-context timeouts `5000`, `scripts/migrate.js` `HARDCODED_EXCLUDES`, and `.pi/agents` target dir). Two prior artifact chains are directly relevant: `thoughts/shared/research/2026-04-12_11-57-02_advisor-settings-persistence.md` establishes `~/.config/rpiv-pi/<plugin>.json` as the persistence convention, and `thoughts/shared/designs/2026-04-13_17-00-00_extract-rpiv-plugins.md` **contradicts** it by moving each sibling's config to `~/.config/rpiv-<plugin>/` — a scope decision this research must resolve. `ctx.ui.*` offers `notify`, `confirm`, `select`, `input`, `custom<T>`, `editor`, `setWidget` per `@mariozechner/pi-coding-agent` types.d.ts, but only `notify` and `confirm` are exercised today — building this command is the first `ctx.ui.custom<T>` / multi-field-picker surface in the repo. Key files read for depth: `index.ts`, `guidance.ts`, `git-context.ts`, `package-checks.ts`, `agents.ts`, `package.json`, and the advisor-settings-persistence research doc.

## Questions

1. **Tracing the `display: false` toggle surface.** The user's canonical example is "change `display: false | true` inside `index.ts`". Trace every occurrence: `pi.sendMessage({ customType: "rpiv-git-context", display: false })` at `extensions/rpiv-core/index.ts:57` (session_start) and `:94` (session_compact); the returned `BeforeAgentStartEventResult.message` with `display: false` at `index.ts:117-119`; and the two `pi.sendMessage({ customType: "rpiv-guidance", display: false })` calls at `guidance.ts:147-151` and `guidance.ts:187-191`. Also follow `takeGitContextIfChanged()` at `git-context.ts:68-75` (produces the string) and `handleToolCallGuidance()` at `guidance.ts:162-192` (produces the multi-file content) into those sites. What does each `customType` mean to the Pi transcript renderer, what would a user actually see if any or all flipped to `true`, and is the correct knob one global `showInjections` boolean or two per-namespace toggles (`showGitContext`, `showGuidance`)? The question matters because this is the user's explicit v1 target and drives the settings schema.

2. **Persistence location and the per-plugin-config contradiction.** The advisor-settings-persistence research at `thoughts/shared/research/2026-04-12_11-57-02_advisor-settings-persistence.md` established `~/.config/rpiv-pi/<plugin>.json` as the convention, with `loadConfig()`/`saveConfig()` modeled on the since-extracted `extensions/web-tools/index.ts:32-56`. But the extract-rpiv-plugins design at `thoughts/shared/designs/2026-04-13_17-00-00_extract-rpiv-plugins.md` moved each sibling's config out of that shared directory (e.g. `~/.config/rpiv-advisor/advisor.json`). Where should `settings-rpiv-pi`'s config live — `~/.config/rpiv-pi/rpiv-core.json` (global scope matching the original), `<cwd>/.rpiv/settings.json` (per-project so `thoughts/` path overrides stay with the repo), or a layered merge — and how does that choice interact with `PACKAGE_ROOT` resolution at `agents.ts:19-23` and the read-only `PI_AGENT_SETTINGS = ~/.pi/agent/settings.json` at `package-checks.ts:15`? This decides whether overrides are user-wide, repo-wide, or both.

3. **Command handler shape, contrasted with `/rpiv-setup`.** The only multi-step UI command in the repo is `/rpiv-setup` at `extensions/rpiv-core/index.ts:140-242`: it guards on `ctx.hasUI` (line 143), builds a list of items (`missing`), calls `ctx.ui.confirm(title, body)` once (line 197), and loops `pi.exec()` with progress `ctx.ui.notify()` (lines 206, 239). Trace that shape and identify where `/settings-rpiv-pi` diverges: does it need a live panel that stays open while the user toggles multiple fields (requires `ctx.ui.custom<T>` from `ExtensionUIContext` at pi-coding-agent types.d.ts:93, currently unused in this repo), or is a sequence of `ctx.ui.confirm` / `ctx.ui.select` / `ctx.ui.input` calls enough? Where should the handler live — inline like `rpiv-setup` or in a new `settings.ts` module following the utility-module rule enforced by `guidance.ts:14-19` (no `ExtensionAPI` in pure utilities, but command handlers DO take `ctx`)?

4. **Rendering the settings panel via `ctx.ui.custom<T>` + pi-tui primitives.** The repo's `ctx.ui.*` usage today is only `notify` (9 sites) and `confirm` (1 site at `index.ts:197`) — no `select`, `input`, `custom`, `editor`, or `setWidget` anywhere. The architecture guidance at `.rpiv/guidance/extensions/rpiv-core/architecture.md` references `TodoOverlay` and `setWidget(KEY, factory, { placement: "aboveEditor" })` as the persistent-widget precedent, but that module was extracted out — only the pattern description remains. Trace how a `custom<SettingsResult>(factory)` call would compose `Container`, `SelectList`, `Text` from `@mariozechner/pi-tui`, how keyboard events flow back to resolve the promise, and whether the overlay should be modal (`overlay: true` option) or in-line. This matters because building the first such widget here is net-new infrastructure and sets a template for future settings-surface commands.

5. **Cache-invalidation propagation when a setting changes at runtime.** The extension maintains three in-memory state shards: the `injectedGuidance` Set at `guidance.ts:104` (cleared by `clearInjectionState()` at `:106-108`), the `cache`/`lastInjectedSig` pair at `git-context.ts:17-20` (cleared by `clearGitContextCache()` at `:28-30` and `resetInjectedMarker()` at `:61-63`), and session handlers at `index.ts:38,87,99,106,114` that read settings implicitly. If the user toggles `showGitContext` or changes the tool-name filter list `["read", "edit", "write"]` at `guidance.ts:167`, or the `thoughts/` scaffold list at `index.ts:43-49`, which caches must be invalidated and which hooks must re-run synchronously so the next `tool_call` / `before_agent_start` sees the new value? Should `/settings-rpiv-pi` synthesize a mini "session_compact" (call all three clear functions + `injectRootGuidance`) on save, or require a session restart for non-trivial toggles?

6. **Ranked scope and schema for v1.** The locator surfaced ten candidates — `display` on injected messages (5 sites), `thoughts/shared` base (`index.ts:43-49`), `.rpiv/guidance/architecture.md` path (`guidance.ts:77,123`), `copyBundledAgents` overwrite default (`index.ts:61`), notification verbosity (`ctx.ui.notify` × 9 sites), guidance tool-name filter (`guidance.ts:167`), install timeout `120_000` (`index.ts:208`), git-context timeout `5000` × 3 (`git-context.ts:36-45`), migrate exclusion list (`scripts/migrate.js:27-30`), `.pi/agents` target (`agents.ts:46`). Trace each candidate to its read site and decide: which belong in v1, which need Typebox schemas vs raw JSON, and how does the schema plug into `ctx.ui.custom` rendering? Does the settings schema live alongside the `Type.Object` schemas already used by sibling `pi.registerTool` calls (per architecture guidance), or in a new `settings-schema.ts`? This scopes the initial implementation to a shippable cut.

7. **Headless / non-UI fallback and stdout/stderr discipline.** The `/rpiv-setup` command at `index.ts:143-145` shows the precedent: `if (!ctx.hasUI) { ctx.ui.notify("/rpiv-setup requires interactive mode", "error"); return; }` — but `ctx.ui.notify` in headless mode is opaque. For `/settings-rpiv-pi` the user may want `/settings-rpiv-pi display=true` (positional-arg form, no UI) or `/settings-rpiv-pi --show` (read-only print). Trace how `_args: string` reaches the handler (signature from `@mariozechner/pi-coding-agent` types.d.ts:732 `RegisteredCommand.handler`), how to parse it robustly, and whether to support a read-mode that prints the current resolved config even when `hasUI` is true. This matters because a settings command used in scripted sessions or CI needs to work without `ctx.ui.custom`.
