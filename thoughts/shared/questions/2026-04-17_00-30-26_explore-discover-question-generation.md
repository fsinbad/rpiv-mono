# Research Questions: Incorporate discover-style question generation into `explore`

## Discovery Summary

Five parallel discovery agents mapped both the rpiv-pi skills in question and the rpiv-skillbased reference implementation. `skills/explore/SKILL.md` (262 lines) is today a monolithic solution-analysis skill with role-based Step 2 dispatch (`codebase-locator`, `codebase-analyzer`, `codebase-pattern-finder`, `thoughts-locator`, optional `web-search-researcher`) and outputs `thoughts/shared/solutions/`. `skills/discover/SKILL.md` (242 lines) is the Phase 1 question-generation skill with a distinctive "dispatch slices, not themes" decomposition block at lines 33-94 that no other rpiv-pi skill reuses. `skills/research/SKILL.md` is the downstream consumer — it reads questions artifacts and dispatches `codebase-analyzer` agents with dense question paragraphs as prompts, grouping by file-overlap. The `design` skill declares acceptance of either research OR solutions artifacts in its frontmatter (`skills/design/SKILL.md:3`) but its body (lines 29-46, 394) only handles research — the explore→design seam is advertised but not implemented. Historical context: the two-phase research split was motivated by context-saturation, discovery-informs-analysis, cognitive-load reduction, and checkpoint-placement (documented in rpiv-skillbased/thoughts/shared/designs/2026-04-06 and 2026-04-07). The v0.10.0 simplification (rpiv-skillbased, 2026-04-11) explicitly chose to keep `research-solutions` (explore's predecessor) as a standalone skill feeding `design` directly. The precedent most relevant to the built-in variant is `thoughts/shared/research/2026-04-11_07-47-54_design-iterative-question-subagents.md`, which replaced design Step 3 in-place with 6 parallel dimension subagents. The explicit "explore is non-linear" decision at `thoughts/shared/designs/2026-04-15_23-52-12_artifact-organization-strategy.md:20, 32` is load-bearing: it determines whether chain or built-in integration is architecturally compatible.

## Questions

1. Trace how `explore`'s current Step 2 agent spawn at `skills/explore/SKILL.md:39-44` works today — role-based dispatch of `codebase-locator`, `codebase-analyzer`, `codebase-pattern-finder`, `thoughts-locator`, and optionally `web-search-researcher` — and compare it to `discover`'s decomposition framework at `skills/discover/SKILL.md:33-94`, which mandates "dispatch slices, not themes" (good slices = one capability + one search objective + 2-6 anchor terms; bad slices = broad themes like "plugin stacking"). Each agent in `explore` today receives a role but no sliced topic, so a multi-area feature like "extract X tool + add permissions + sync docs" produces overlapping sweep rather than targeted per-seam depth. Analyze whether porting the slice-decomposition block verbatim (lines 35-94 including dispatch-budget heuristic at line 88-89) into explore would tighten evidence quality for the "Current State" and "Relevant patterns" sections the solutions artifact requires at `skills/explore/SKILL.md:124-133`.

2. Trace what a **chained** variant looks like — `explore` reading a `thoughts/shared/questions/YYYY-MM-DD_HH-MM-SS_*.md` artifact as its argument (mirroring `skills/research/SKILL.md:15-46` Step 1: read artifact FULLY, extract Discovery Summary + dense question paragraphs + frontmatter tags; `skills/research/SKILL.md:51-92` Step 2: group questions by ≥2 shared file references and dispatch analysis agents with the question paragraph as prompt). Analyze how explore would repurpose the existing `agents/codebase-analyzer.md` (tools: `read, grep, find, ls`) for per-question depth analysis, how the current explore Step 3 solution-option generation at `skills/explore/SKILL.md:46-57` would consume per-question findings to populate 2-4 approaches, and whether `skills/discover/SKILL.md:220` should gain an alternative chain suggestion — "run `/skill:research` OR `/skill:explore` with this artifact" — vs. keeping discover/research as the sole chain.

3. Trace a **built-in** variant where explore internally generates and dispatches questions per the design-iterative-question-subagents precedent at `thoughts/shared/research/2026-04-11_07-47-54_design-iterative-question-subagents.md:22-25`, which replaced design Step 3 in-place with 6 parallel dimension subagents (data model / API / integration / scope / verification / performance) each self-prioritizing to top 1-2 questions. For `explore` the dimensions differ because the output is option-comparison not single-architecture: candidate dimensions include approach-shape (what is the implementation pattern), precedent-strength (what similar code exists), integration-risk, migration-cost, verification-cost, and novelty/unknowns. Analyze which explore step absorbs the new pass — before or after Step 3 option-generation — and how it mechanically merges with `skills/explore/SKILL.md:46-57` where 2-4 options get enumerated with pros/cons/complexity.

4. Trace the solutions artifact's body sections at `skills/explore/SKILL.md:75-229` — `Problem Statement` (Requirements/Constraints/Success), `Current State` (Existing implementation, Relevant patterns, Integration points), `Solution Options` (each with How it works / Pros / Cons / Complexity), `Comparison` table, `Recommendation`, `Scope Boundaries`, `Testing Strategy`, `Open Questions`. Each section has a different information source — `Current State` and `Integration points` come from code analysis, `Solution Options` from synthesis across options, `Open Questions` from what remained unresolved. Analyze which sections are meaningfully improved by question-driven dispatch (traces naturally populate `Current State`/`Integration points` because they name specific code paths) vs. which require orthogonal option-enumeration logic that question-generation doesn't help (`Comparison` table, `Recommendation`). This determines whether the new pass should feed into Step 2 (pre-option generation) or Step 3 (per-option depth).

5. Trace the advertised-but-unimplemented `explore → design` seam. `skills/design/SKILL.md:3` frontmatter says "a research artifact from discover → research, or a solutions artifact from explore" — but `skills/design/SKILL.md:29-46` (Step 1 "Read research artifact") and `skills/design/SKILL.md:394` ("requires a research artifact produced by the research skill") only handle research. Meanwhile `skills/research/SKILL.md:210-250` produces sections design consumes (Code References, Integration Points, Architecture Insights, Developer Context, Open Questions) that explore's template lacks. Analyze whether migrating explore to emit a research-compatible payload (either by accepting questions and running research's dispatch, or by generating questions internally and grouping their findings into Code References/Integration Points) would close the advertised gap and let `/skill:design solutions-artifact.md` actually work today, vs. the alternative of editing design Step 1 to parse solutions-format artifacts directly.

6. Trace the explicit non-linear decision for explore in `thoughts/shared/designs/2026-04-15_23-52-12_artifact-organization-strategy.md:20, 32` ("`explore` stays project-level at `{base}/solutions/` and does NOT participate in feature-folder `_meta.md` contract because 'solutions are non-linear'") and the slug-collision logic at line 109 (`discover` derives feature slug; downstream skills extract slug from upstream path argument). Analyze whether a chained variant (explore consuming a questions artifact) would drag explore into feature-scoped linearity — since the upstream questions artifact path already carries a slug — versus a built-in variant that keeps explore's free-text entry point and produces a standalone solutions artifact. This is the load-bearing design decision that determines whether the two integration strategies diverge only in mechanics or in project-layout contract.

7. Trace how question-anchored dispatch (each agent receives a dense question paragraph like `skills/research/SKILL.md:56-85` prescribes) shifts `explore`'s Step 3 "Generate and compare solution options" at `skills/explore/SKILL.md:46-57` and the recommendation rationale at `skills/explore/SKILL.md:58-62`. Today, role-agents return broad dumps and the orchestrator freely enumerates 2-4 options; question-anchored dispatch returns per-question findings with `file:line` evidence, which naturally maps to option-option comparison only if questions were generated with "what's the pattern for X vs Y" framing. Analyze whether explore needs a new question-generation rubric specific to option-comparison (e.g., "for each candidate approach, what precedent exists and what breaks?") distinct from discover's trace-quality rubric at `skills/discover/SKILL.md:114-136` (which is tuned for single-architecture depth, not multi-option divergence).
